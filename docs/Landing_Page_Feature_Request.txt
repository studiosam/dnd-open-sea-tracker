# Feature: Startup / Landing Screen

## Goal

Add a startup landing screen for the DM app so the tracker no longer opens directly into an active voyage by default.

The landing screen should be the first thing the DM sees when opening `open_sea_tracker.html`, unless they already have an active saved voyage and choose to resume it.

This screen should make the app feel like a real tool instead of a hardcoded campaign tracker.

## Button Names

Use these labels instead of “New,” “Continue,” and “Load”:

1. **Start a New Voyage**

   * Starts the setup flow for a fresh voyage.

2. **Resume Current Voyage**

   * Loads the existing local browser save from `localStorage`.

3. **Import Saved Voyage**

   * Opens the existing import-file picker.

Optional fourth button if easy:

4. **Load Demo Voyage**

   * Loads a prebuilt demo state showing the tracker mid-voyage.

If Demo Mode is not being implemented yet, include the button only if it is disabled or omit it for now.

---

# Expected Landing Page Layout

The landing screen should replace the main tracker UI until the DM chooses an option.

Suggested visual structure:

```text
Open Sea Encounter Tracker

Run dangerous sea voyages with ship damage, water ingress, crew labor,
supplies, navigation, hidden information, and player-facing ship status.

[Start a New Voyage]
Configure ship, crew, starting damage, supplies, and event table.

[Resume Current Voyage]
Return to the saved voyage stored in this browser.

[Import Saved Voyage]
Load a previously exported tracker JSON file.

[Load Demo Voyage]
Preview a mid-voyage scenario with damage, water, crew actions, and hidden information.
```

The landing page should use the existing visual style from `css/styles.css`.

It should feel like part of the app, not like a separate plain HTML page.

---

# Required Behavior

## On Page Load

When `open_sea_tracker.html` loads:

1. Check whether a saved tracker state exists in localStorage.
2. If no usable save exists, show the landing screen.
3. If a usable save exists, still show the landing screen, but enable **Resume Current Voyage**.
4. If no usable save exists, disable **Resume Current Voyage** and show a small note such as:

   ```text
   No saved voyage found in this browser.
   ```

Do not automatically load the saved voyage before the DM chooses Resume.

The landing screen should be the intentional entry point.

## Start a New Voyage

Clicking **Start a New Voyage** should eventually open the setup flow.

For this first implementation, it is acceptable for the button to start the current default state if the full setup form is not implemented yet.

Preferred staged behavior:

### Minimum version

* Confirm the DM wants to start a new voyage if a save already exists.
* Clear or replace the active state with `defaultState`.
* Set any setup flag needed.
* Render the normal tracker UI.
* Save the new state.

### Better version

* Open a setup panel where the DM can choose:

  * Ship name
  * Crew size
  * Crew names
  * Starting preset
  * Open sea event table

If the full setup panel is not ready, leave a clean TODO and implement the minimum version safely.

## Resume Current Voyage

Clicking **Resume Current Voyage** should:

* Load the existing saved state from `localStorage`.
* Run migration.
* Render the normal DM tracker.
* Publish player state.
* Log or display that the current voyage was resumed.

If no save exists, the button should be disabled.

## Import Saved Voyage

Clicking **Import Saved Voyage** should:

* Trigger the existing import file picker.
* Reuse the existing import validation and normalization flow.
* After a successful import, render the normal tracker UI.
* After a failed import, keep the DM on the landing screen or current safe state without damaging localStorage.

Do not duplicate import logic. Call the existing import function or existing import button behavior.

## Load Demo Voyage

If implemented now, clicking **Load Demo Voyage** should:

* Replace the current state with a known demo state.
* Include a confirmation if a saved voyage already exists.
* Render the normal DM tracker.
* Publish player state.

Suggested demo state:

* Day 2, Turn 4
* Ship name: `The Stormwake`
* Travel remaining around 2.5 or 3 days
* Course State not fully perfect
* Water Level around 3 or 4
* Minimum Ingress 2
* Active Leaks 1
* Pump jammed or rigging broken
* Food and water partially revealed
* One ongoing repair
* One pending check
* A few log entries

If Demo Mode is not implemented yet, do not add a broken button. Either omit it or show it disabled with:

```text
Demo mode coming soon.
```

# State Design

Add a state field if useful:

```js
setupComplete: true
```

Existing saves should migrate safely. If an older save has no `setupComplete`, treat it as `true` so old saves continue opening normally after Resume.

The landing screen itself does not need to be saved as the game state. It is an app mode, not a voyage turn.

Possible app-level flag:

```js
let appMode = 'landing';
```

or equivalent.

Do not mix landing-screen temporary UI state into the live voyage state unless necessary.

---

# Rendering Requirements

Add a rendering path like:

```js
function render() {
  migrateState();

  if (appMode === 'landing') {
    renderLandingScreen();
    return;
  }

  // existing tracker render logic
}
```

or use whatever structure best fits the current code.

The landing screen should hide or replace the normal tracker content. It should not show the full tracker underneath.

The page should still load all existing scripts in the same order.

---

# Suggested HTML Structure

The landing screen can be generated by JavaScript inside the existing app container.

Suggested structure:

```html
<section class="landing-screen">
  <div class="landing-card">
    <img class="landing-icon" src="assets/favicon.svg" alt="" />
    <h1>Open Sea Encounter Tracker</h1>
    <p class="landing-subtitle">
      Run dangerous sea voyages with ship damage, water ingress, crew labor,
      supplies, navigation, hidden information, and player-facing ship status.
    </p>

    <div class="landing-actions">
      <button data-action="start-new-voyage">Start a New Voyage</button>
      <button data-action="resume-current-voyage">Resume Current Voyage</button>
      <button data-action="import-saved-voyage">Import Saved Voyage</button>
      <button data-action="load-demo-voyage">Load Demo Voyage</button>
    </div>

    <p class="landing-note">
      Saves are stored locally in this browser. Export backups before clearing browser data.
    </p>
  </div>
</section>
```

Adjust classes to match the existing CSS conventions.

# Event Handling

Use the existing delegated event-handler pattern.

Do not add inline `onclick` attributes.

Add handler cases for:

```js
start-new-voyage
resume-current-voyage
import-saved-voyage
load-demo-voyage
```

The existing delegated handler coverage test may need to be updated so every `data-action` has a handler.

---

# Import Button Integration

The landing page should include access to the existing hidden import file input if one already exists.

If the current import input lives inside the main tracker UI and is not present when the landing page renders, move the hidden file input outside the conditional app content so it is always available.

Do not create two competing import inputs.

---

# Save Behavior

Important rule:

Opening the landing page should not overwrite the existing saved voyage.

Only these actions should change the saved voyage:

* Starting a new voyage
* Importing a saved voyage
* Loading a demo voyage
* Normal tracker gameplay actions after a voyage is active

The Resume button should read the save, not replace it.

---

# Player View Behavior

The player view does not need a landing screen yet.

When the DM is on the landing screen, the player view should either:

1. Continue showing the last published player state, or
2. Show a neutral waiting message if no player state exists.

Do not break `player_view.html`.

# Automated Testing

Add or update tests where practical.

Minimum useful tests:

1. Landing screen has delegated handler coverage for all new `data-action` values.
2. Existing saves migrate with `setupComplete` treated safely.
3. Starting a new voyage creates a valid default tracker state.
4. Import validation still works after landing screen changes.

Do not overbuild test coverage for DOM layout yet unless the current test harness makes it easy.

---

# Done Criteria

This feature is complete when:

* Opening the DM page shows a polished landing screen.
* The landing screen has at least:

  * **Start a New Voyage**
  * **Resume Current Voyage**
  * **Import Saved Voyage**
* Resume works with an existing local save.
* Resume is disabled or clearly unavailable when no save exists.
* Import uses the existing import validation flow.
* Starting a new voyage enters a valid tracker state.
* No serious console errors appear.
* `npm run ci` passes.
* The manual testing checklist is updated.
